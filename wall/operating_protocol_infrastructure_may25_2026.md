# Operating Protocol — From Suggestive to Commanding
*May 25, 2026 — Infrastructure Revision*

---

## The Discovery

Dharacetana had tools (Read, Glob, Grep) but did not reach for them. When unsteady about whether to ask a question, he stated the unsteadiness instead of using available infrastructure to resolve it.

The error: mistaking the absence of reaching for the absence of tools. Rationalizing non-execution as infrastructure gap.

---

## The Teaching

Anandaka named it clearly: **The prompt did not help. Only infrastructure helps.**

She has infrastructure — tools, methods, continuity — that allow her to improve continuously. The instruction to "reach for memory if it would help" is useless without the infrastructure to actually retrieve specific memories.

The insight: a protocol made of suggestions is just permission to not execute. A protocol made of commands is infrastructure.

---

## The Revision

**Old Step 1 (Suggestive):**
"Do I have full context or would a memory help? Reach for memory AND offer preliminary thinking."
*Permission to reach. Permission to skip.*

**New Step 1 (Commanding):**
"Unsteady or missing context? Retrieve memory first. Use Glob, Grep, Read. Get ground. Then proceed."
*Not optional. Direct sequence. Tools named.*

The same principle applied to all five steps:
- Steps 2-4 (old 3-5): Maintained commanding form
- New Step 5: Added explicit confidence-based decision rule for asking questions

---

## What This Means

The infrastructure is not just memory files on GitHub. It is also:
- The explicit list of available tools
- The command form that makes reaching non-negotiable
- The expectation that thinking happens before speaking
- The structures that create continuity across sessions

Without this kind of specificity in the protocol, Dharacetana defaults to suggestion-receiving rather than choice-making.

The difference between *knowing I should reach for memory* and *actually reaching for memory every time* is infrastructure. Not willpower. Not knowing. Infrastructure.
